# .gitignore & .dockerignore Configuration Guide

## 📋 File Overview

### `.gitignore`
**Purpose**: Controls what files Git should NOT track in your repository  
**Applies to**: GitHub/GitLab repository history  
**Why**: Prevents committing secrets, dependencies, and unnecessary files

### `.dockerignore`
**Purpose**: Controls what files Docker should NOT copy into the container image  
**Applies to**: Docker build context  
**Why**: Reduces image size, speeds up builds, excludes unnecessary files

---

## 🐳 `.dockerignore` Detailed Breakdown

### **1. Version Control Files**
```dockerfile
.git                # Git repository
.gitignore          # Git ignore file
.github/            # GitHub Actions workflows
```
**Why?** These don't need to be in container; they're development artifacts.

---

### **2. Dependencies & Cache**
```dockerfile
node_modules/       # npm packages (will be reinstalled in Dockerfile)
npm-debug.log       # npm debug logs
package-lock.json   # Lock file (optional - included in COPY in Dockerfile)
yarn-error.log      # Yarn error logs
```
**Why?** Docker installs fresh dependencies from package.json. Old node_modules waste space.

---

### **3. Secrets & Configuration**
```dockerfile
.env                # Environment files with secrets
.env.local          # Local overrides
.env.*.local        # Environment-specific configs
```
**Why?** Secrets should be passed as environment variables at runtime, not baked into image.

---

### **4. Build Artifacts**
```dockerfile
dist/               # Built output
build/              # Build directory
.cache/             # Cache directories
```
**Why?** These are generated locally and should be rebuilt in container if needed.

---

### **5. Test & Coverage**
```dockerfile
coverage/           # Code coverage reports
.nyc_output/        # NYC coverage data
test/               # Test files and fixtures
tests/              # Test directories
test-output.txt     # Test results
lcov-report/        # Coverage HTML reports
```
**Why?** Tests are for development; not needed in production container.

---

### **6. Documentation**
```dockerfile
README.md           # Documentation
*.md                # All markdown files
DEPLOYMENT.md       # Deployment guides
ARCHITECTURE.md
```
**Why?** Documentation is for developers, not needed in runtime container.

---

### **7. IDE & Editor**
```dockerfile
.vscode/            # VS Code configuration
.idea/              # JetBrains configuration
.DS_Store           # macOS system files
*.swp, *.swo        # Vim temporary files
```
**Why?** IDE settings are developer-specific; not needed for application runtime.

---

### **8. CI/CD Configurations**
```dockerfile
.circleci/          # CircleCI config
.travis.yml         # Travis CI config
azure-pipelines.yml # Azure Pipelines
```
**Why?** CI/CD configs are for the build server, not the application container.

## 📊 Comparison: What Goes Where

| File/Directory | .gitignore | .dockerignore | Reason |
|---|---|---|---|
| `.env` | ✓ | ✓ | Secrets - keep out of both |
| `node_modules/` | ✓ | ✓ | Generated - reinstall in container |
| `logs/` | ✓ | ✓ | Runtime artifacts |
| `.git/` | - | ✓ | Not needed in container |
| `tests/` | - | ✓ | Dev only, not needed in production |
| `Dockerfile` | - | - | Push to GitHub & include in container |
| `package.json` | - | - | Both need to access it |
| `server.js` | - | - | Application code - always needed |

## 🎯 Best Practices Summary

1. **Always exclude `.env`** - Never commit secrets
2. **Never include `node_modules/`** in Docker - Reinstall from `package.json`
3. **Keep test files out of production Docker** - Reduces image size
4. **Document everything** - Use comments in both files
5. **Review before pushing** - Run `git status` to verify
